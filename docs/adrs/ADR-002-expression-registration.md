# ADR 001: Expression registration inside DataStar Attributes:

**Status:** Accepted

**Date:** 2026-03-25

## Context

Some DataStar attributes can receive DataStar Expressions, and the problem is how those expressions are passed,
in order for them to be consumed and used in the attribute.
Also, the support for strongly typed Events, passed to `dataOn`,
and, in the future, for the DataStar variable `el` is needed; with these
being accessible when creating an expression inside an attribute.

There were three previous implementations for this problem:

The first was for the attribute function to receive a lambda that was an extension function of a builder.
This builder provided functions that internally registered the expression passed, and the attribute instantiates its
class, applies the lambda received, and generates the final expression. There was a `CodeBuilder`, a `ModBuilder`
and a `ModCodeBuilder`. The first was for attributes that only received `DataStarExpression`s, without modifiers,
the second was the inverse, and the last when both code and modifiers could be passed.
`CodeBuilder` exposed the function `code(block: () -> String)` to register expressions; `ModBuilder` exposed
`mods(block: M.() -> Unit)` to register modifiers, allowing only those of the specific attribute to be called,
and `ModCodeBuilder` exposed both.
This was then rejected as it introduced new vocabulary to the DSL, which was unwanted.

The second was to create an `open class DataStarExpression internal constructor(val expression: String)`
that represents anything that is a DataStar expression. With this, both the `Signal` and the `DataStarAction` classes
extended it, allowing for expressions to be built and passed directly as a parameter of attribute functions,
like so: `fun <...>dataInit(expr: DataStarExpression)`.

The third came to be as a response to the need to strongly type, and make available for the expressions,
the events that are received in the dataOn attribute. With this, the second approach was no longer sufficient.
With this, we took a somewhat step backwards to the first approach, but CodeBuilder was renamed to ExpressionBuilder,
`ModBuilder` to `ModifierBuilder`, `ModCodeBuilder` to `ExpressionModifierBuilder`, and a new `EventExpressionBuilder`, with
a property of the event passed onto the attribute.
The `mods()` function was also renamed to `modifiers()`, but the most significant change was the shift from the
function `code()` to the use of the unaryPlus operator to register expressions. This allowed for a more interpretive way
of adding expressions, without the addition of new vocabulary.

## Decision
The decision taken was to move the functions that generate `DataStarExpression`s, such as actions and infix functions,
to the `ExpressionBuilder` class. This enforces that they can only be called inside the lambdas received by attributes.
It also makes the `builderExpression` property available to these functions, enabling them to directly add their respective
expressions to the accumulator. This makes it possible to remove the unaryPlus operator for registering expressions
generated by actions and other functions of the DSL.
The unaryPlus operator was maintained to pass raw strings or `Signal`s to an attribute, acting as a mechanism to
maintain flexibility.

### Expression Accumulator Model:

Expressions are constructed eagerly during the execution of the DSL lambda.
- `ExpressionBuilder` maintains a mutable, ordered list (`builderExpression`) of `DataStarExpression`s.
- DSL functions (such as actions and operators) directly register their resulting expressions into the accumulator.
- The accumulator may, temporarily, contain partial expressions during evaluation.
- Functions that receive two operands (such as an infix function) perform normalization by:
    - identifying previously registered operand expressions,
    - removing them from the accumulator,
    - inserting the new expression representing the composed result.
- Some operands (such as `Signal`s or Strings) may not be registered in the accumulator prior to participating in
operations.

This model prioritizes DSL ergonomics (removal of the explicit registration operator for operations and actions)
at the cost of introducing controlled mutation and normalization logic within the accumulator.

## Trigger

Remove the necessity of calling the `unaryPlus` operator to register operations supported directly with the DSL functions while maintaining support for `evt` and, in the future, `el` DataStar variables.

## Consequences

**Positive:**
- Clean way to process operations, with clean division where, signals and strings are processed with the
unaryPlus operator, and other operations register the expressions directly, removing the need for the
unaryPlus.

**Negative:**
- Increased complexity of the internal implementation of the expression accumulator, going from a simple
`StringBuilder` to a `MutableList<DataStarExpression>` with multiple additions and removals from the list.
    - Each function adds its expression directly to the internal accumulator, this poses a problem
when calling an infix function, namely the order of processing and registering of expressions.
An example is as follows:
        ```kotlin
        dataOn(Click) {
            !fetching and get(::loadMore)
        }
        ```
        In this case, the not operator registers the not plus fetching signal, then the get action is registered, and only
        after does the infix function `and` execute; and multiple combinations of types can be received in these
        infix functions, like Strings or Signals that, by themselves, do not register in the accumulator.
        To solve this, the chosen approach is that operator and action functions, always register their expressions.
        The infix operations then check if the parameters that are `DataStarExpression`s are `Signal`s or not,
        and if they are not, first remove them from the accumulator, and then adds the final expression
        altogether. This ensures that non-`DataStarExpression`s are always registered when used in concatenation
        with the infix functions, even without the use of the unaryPlus operator, but that the accumulator stays clean
        and consistent after each operation.
- The model for accumulation of expressions can lead to bugs due to its constant add/remove structure

**Neutral:**
- Strings and Signals are processed the same way and the attributes still receive a lambda with the expressions.

## Alternatives considered

### DataStarExprElem

An alternative with an additional class `DataStarExprElem` was considered. It represented an element of the `DataStarExpression`,
where the latter would only represent the final expression of the attribute. The element class would then be specialized
by `DataStarSimpleExprElem`, representing the result of a simple expression, and `DataStarCompoundExprElem`, the
culmination of two or more expressions.
This had the objective of better characterizing the expressions in the accumulator to try and reduce the number of
operations done in the list.
It was deemed that the structural complexity it introduced was not worth the benefits it was trying to achieve,
and that it was an unnecessary addition as the `DataStarExpression` class was, in practice, useless, as it was only
instantiated when calling the final expression where, at that point, whether it was a `DataStarExpression` or just the resulting String
was not important anymore. Due to this it was discarded.
