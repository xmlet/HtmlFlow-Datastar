# ADR-004: DataStarExpression restructure

**Status:** Accepted

**Date:** 2026-04-30

## Context

With the introduction of the `DataStarExpression` class, both `Signal` and
`DataStarAction` classes extended it, but, due to significant additional
logic that they had, a clear separation between all of this was still kept.

As the value property was removed from `Signal` and the action functions
were moved to the `ExpressionBuilder` class, neither of them held any
additional logic anymore.

## Decision

Make `DataStarExpression` a sealed class with `Signal`, `DataStarAction`
and an additional `DataStarExpressionOp` (expressions generated by the DSL
functions, such as the JavaScript operators) extending it.

## Trigger

Neither `Signal` nor `DataStarAction` having any additional expression logic.

## Consequences

**Positive:**

- Clearer separation and definition of roles;
- Access to compile time exhaustive when expressions without else branch.

**Negative:**

- Creation of an additional class.

**Neutral:**

- `DataStarExpression` is now sealed and can no longer be instantiated directly.
Direct instantiations were replaced with `DataStarExpressionOp`.

## Alternatives Considered

No other alternatives were considered.
